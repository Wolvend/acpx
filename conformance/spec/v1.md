# ACP Conformance Profile v1 (Draft)

## Purpose

Define a minimal, stable conformance profile for ACP transport/session behavior
that can be validated across adapters.

This profile intentionally focuses on the highest-leverage interoperability
surface first.

## Normative language

The key words "MUST", "MUST NOT", "SHOULD", and "MAY" are to be interpreted as
RFC 2119 terms.

## Transport assumptions

- Transport is JSON-RPC 2.0 over stdio streams.
- Requests use unique client-generated `id` values.
- Notifications do not include `id`.

## Required methods in profile v1

- `initialize`
- `session/new`
- `session/prompt`
- `session/cancel`

Adapters under test MUST also emit `session/update` notifications during prompt
execution and completion.

## Contract requirements

1. `initialize`

- Adapter MUST respond with a valid JSON-RPC success response.
- Response MUST include a protocol version value.
- If capabilities are returned, they MUST be an object.

2. `session/new`

- Adapter MUST return a non-empty session identifier.
- If session capabilities are returned, they MUST be an object.

3. `session/prompt`

- Adapter MUST accept a valid content block payload.
- Adapter MUST emit at least one `session/update` notification before terminal
  completion for non-empty prompt execution.

4. `session/update`

- Update notifications MUST reference an existing session id.
- Completion semantics MUST be observable via a terminal state or final
  stop reason in update stream.

5. `session/cancel`

- For an active turn, adapter MUST acknowledge cancel request via success
  response or documented cancellation acknowledgement.
- The active turn MUST transition to a cancelled terminal state (or equivalent
  adapter-defined cancelled reason).

6. Error semantics

- Invalid request params MUST yield a JSON-RPC error (typically `-32602`).
- Operations for unknown session ids MUST yield an explicit error (not silent
  success).
- Permission denials triggered by client policy MUST be represented as explicit
  failures with machine-readable code/message.

## Timing guidance

- Each case defines:
  - `request_timeout_ms`
  - `update_timeout_ms`
  - optional `settle_timeout_ms`
- Runner SHOULD treat timeout as a test failure with captured transcript.

## Case file schema (runner-facing)

Each case JSON object includes:

- `id`: stable case id
- `title`: human-readable title
- `profile`: conformance profile id (`acp-core-v1`)
- `description`: goal of the case
- `permission_mode` (optional): per-case override (`approve-all` or `deny-all`)
- `steps`: ordered operations consumed by the runner engine
- `checks`: required outcomes evaluated after steps
- `timeouts`: timeout budget per case

## Pass/fail model

- A case passes only if all assertions pass.
- A profile run passes only if all required cases pass.
- Optional cases MAY be reported separately without failing the profile.

## Out of scope for v1

- `session/fork`
- `session/list`
- `session/resume`
- `session/set_model`
- `$/cancel_request`

These can be introduced in later profile revisions once their semantics are
stable across adapters.
